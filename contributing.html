Introduction

As an open-source project, the Readium Foundation welcomes contributions from everyone - from individuals, groups and corporations. The best way to get started with Readium is to read through the introductory materials on this site, subscribe to the mailing list and, of course, get the source code and start spelunking through it.

An important note to bear in mind is that YOU are the Readium team. Readium has no "dedicated team". Other than a consultant or two and a couple of part-time employees, everything in Readium is done by the Readium contributors - some employed by Readium Foundation members, some simply individual contributors.

Contributing to the Readium Project

Looking to contribute something to the Readium project? Here's how you can help:

Reporting Issues

Please read the following guidelines before opening any issue.

  1. Ensure that your EPUB is valid. Always validate your EPUB against EPUBCheck. If there are errors or warnings, please fix those before submitting an issue.
  2. Search for existing issues. We get a lot of duplicate issues, and you'd help us out a lot by first checking if someone else has reported the same issue. Moreover, the issue may have already been resolved with a fix available.
  3. Create an isolated and reproducible test case. Be sure the problem exists in Readium (try your HTML in the browser) code & samples with a reduced test case that should be included in each bug report.
  4. Include a live example. Ideally with an EPUB, if appropriate or a clear code sample.
  5. Share as much information as possible. Include operating system and version, browser and version, version of the SDK/Chrome Extension as appropriate.
  6. Include ALL steps to reproduce the bug.

Finally, Readium is a complex project with many parts (repos). Try to ensure that you post the issue to the right repo. In general, bugs in the Chrome Extension or CloudReader are in readium-js-viewer. Bugs that are Core issues in the SDK of course go to readium-sdk. If in doubt, contact Ric Wright for guidance.

Pull Requests

  • Try not to pollute your pull request with unintended changes--keep them simple and small
  • Try to share how you have tested your code before submitting a pull request. Provide as much detail as possible.
  • Pull requests should always be created against the develop branch - never the master.

Coding Standards

The following guidelines are aimed at self-created code. Clearly, if the code is an EPUB and was created by a tool such as InDesign then these guidelines might not apply.

For all code, please remove all tabs, replacing them with 4 spaces. Please set this preference in your tool(s).

C++, C# and Java

  • Readium doesn't have a required style-guide, but the following are good guidelines
  • Google C++ and NYT Objective-C are good for those languages.
  • Similarly, the Google Java and Oracle Java are good for Java.
  • Microsft has a guide for C# coding.
  • If in doubt, please post your questions to the readium-dev mailing list.

JS

  • End all lines with a semi-colon
  • Comma last
  • Make it readable and attractive
  • Please use jsDoc annotations/comments to add typing information.

HTML

  • Double quotes only, never single quotes
  • EPUB must at least for now, be HTML5.
  • The XML serialization of HTML is required in EPUB 3.0. This may change in EPUB 3.1 but for backwards compatibility using the XML serialization is strongly encouraged.
  • Use tags and elements appropriate for an HTML5 doctype (e.g., self-closing tags)
  • Use CDNs and HTTPS for third-party JS when possible. We don't use protocol-relative URLs in this case because they break when viewing the page locally via file://

CSS

  • Adhere to the RECESS CSS property order
  • Multiple-line approach (one property and value per line)
  • Always a space after a property's colon (e.g., display: block; and not display:block;)
  • End all lines with a semi-colon
  • For multiple, comma-separated selectors, place each selector on its own line
  • Attribute selectors, like input[type="text"] should always wrap the attribute's value in double quotes, for consistency and safety (see this blog post on unquoted attribute values that can lead to XSS attacks)
